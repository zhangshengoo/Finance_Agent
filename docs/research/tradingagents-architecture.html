01 / MENTAL MODEL

核心理念：把交易公司编码成 Agent 图

TradingAgents 不追求单个"全能 agent"，而是用职责分工 + 对抗辩论来逼近一支真实买方团队的决策过程。每个角色视角受限、激励不同，冲突在辩论中被显式暴露，再由管理层综合。

"专业分析师采集情报，研究员就投资论点正反交锋，交易员制定方案，风控团队把关仓位，组合经理签发最终交易。"
设计哲学

分工 > 全能

四类分析师各管一个数据维度（行情 / 情绪 / 新闻 / 基本面），只产出本职报告，避免单一上下文里信息互相污染。

对抗机制

辩论挤出偏差

多头 vs 空头、激进 vs 保守 vs 中性——用结构化辩论强制暴露分歧，比"一个 agent 自说自话"更难自我强化偏见。

持续学习

反思即记忆

每笔决策落盘成 Markdown 日志；下次运行回填 5 日实际收益与 alpha，生成教训，再注入未来同票/跨票分析。

02 / ORCHESTRATION

编排流水线：一张 LangGraph 状态图

TradingAgentsGraph 用 LangGraph StateGraph 把所有角色连成有向图。分析师通过工具调用取数、消息清理节点在切换 agent 前清空上下文；两处辩论用计数器而非"收敛判断"来终止，保证可复现。状态对象 AgentState 贯穿全程传递各份报告与辩论历史。

04 / THE DEBATES

两场辩论：对抗是核心引擎

框架最具特色的设计是两段结构化辩论。它们形态不同：多空是二元对抗，风控是三方轮转。共同点是都用消息计数终止、都把分歧写进共享状态供管理层裁决。

辩论一 · 二元对抗

多头 ⇄ 空头

多头与空头交替发言，各读全部分析报告 + 对手上一轮论点，要求"对话式反驳"而非罗列事实。investment_debate_state["count"] 每次 +1，达到 2 × max_debate_rounds 后由研究经理综合判定胜负，写入 investment_plan。

终止count ≥ 2N（默认 N=1 → 4 条消息）
裁决研究经理 → 5 档建议
辩论二 · 三方轮转

激进 → 保守 → 中性

三位风险员循环发言，每人检查 latest_speaker 决定下一棒。risk_debate_state["count"] 达到 3 × max_risk_discuss_rounds 后由组合经理综合三方 + 注入历史教训，签发 final_trade_decision。

终止count ≥ 3N（默认 N=1 → 3 条消息）
裁决组合经理 → 最终评级 + 价位 + 时间窗
调大辩论深度的代价。 将 max_debate_rounds / max_risk_discuss_rounds 从 1 调到 2，会线性增加 LLM 调用与 token 成本，但能让论点交锋更充分——这是质量 / 成本的核心旋钮。
05 / DATAFLOWS

数据流：厂商路由 + 优雅降级

06 / STATE & MEMORY

状态与记忆：从盈亏中学习的反思环

记忆是 append-only 的 Markdown 文件（~/.tradingagents/memory/trading_memory.md）。决策分两相：当次运行写 pending 条目；下次同票运行回填实际收益与反思。组合经理在拍板前被注入历史教训，形成闭环学习。此外 checkpointer.py 用 per-ticker SQLite 支持崩溃续跑。

1

Phase A · 当次落盘（pending）

store_decision() 把最终决策追加为 [日期 | TICKER | 评级 | PENDING] 条目；同时把完整 state 存为 JSON 日志。SignalProcessor 用正则从 Markdown 提取 5 档评级，无需二次 LLM 调用。

2

Phase B · 下次回填收益

_resolve_pending_entries() 拉取 T 与 T+5 价格，算原始收益 (p[T+5]−p[T])/p[T] 与相对基准的 alpha。基准由 benchmark_ticker 或后缀映射决定。

3

反思生成

Reflector 产出 2–4 句反思：方向判断对不对（引用 alpha 数值）、论点哪部分成立/失败、给未来一条具体教训。写回日志条目。

4

注入未来决策

组合经理读取同票近 5 条决策 + 跨票近 3 条教训，嵌入提示词，让每次分析"站在历史经验上"。memory_log_max_entries 可限制条数并裁剪最旧条目。

核心状态对象

AgentState

MessagesState 子类，贯穿全图：company_of_interest · market/sentiment/news/fundamentals_report · investment_debate_state · risk_debate_state · investment_plan · trader_investment_plan · final_trade_decision。

续跑机制

Checkpoint & Resume

开启后用 per-ticker SqliteSaver（~/.tradingagents/cache/checkpoints/<TICKER>.db），thread_id = {ticker}_{date}。同票同日续跑、异日重开；成功完成后自动清理。

08 / DESIGN PATTERNS

值得借鉴的设计模式

这些是框架里防幻觉、保可复现、可降级的关键工程决策——对任何"LLM 接金融数据"的系统都通用。

PATTERN 01 结构化输出 + Markdown 回渲染

研究经理 / 交易员 / 组合经理用 bind_structured() 绑定 Pydantic schema，invoke_structured_or_freetext() 成功则解析、失败则降级自由文本。每个 schema 配 render_*() 把对象转回 Markdown，让下游始终看到一致形状——类型安全 + 向后兼容兼得。

PATTERN 02 情绪分析师的"预取式接地"

不让 LLM 自行调工具，而是先取齐新闻 / StockTwits / Reddit 三源数据作为结构化块注入提示词；结构化输出保证表头（情绪带 + 分数 + 置信度）确定，自由文本叙事保留逐源细节。某源失败返回占位串而非抛错。

PATTERN 03 标的身份解析与锚定

resolve_instrument_identity() 对 ticker 做一次 LRU 缓存的 yfinance 查询，让所有 agent 锚定同一家公司，避免"看着价格走势幻觉成另一家公司"。yfinance 不可用时 fail-open 返回 {} 不阻塞分析。

PATTERN 04 厂商 fallback 链 + 显式哨兵

主厂商 → 其余厂商 → NO_DATA_AVAILABLE。限流错误换厂商、NoMarketDataError 记下继续找、全失败时抛出第一个真实错误（网络/Key）。宁可显式说"没数据"也绝不编造。

PATTERN 05 计数终止 + 消息清理节点

辩论用消息计数而非收敛判断终止，保证可复现、防死循环。分析师切换前的"消息清理节点"删除全部消息并注入占位上下文，让下一个 agent 不带前者的工具调用记忆，干净开场。

一次 NVDA 运行的全貌

以 ta.propagate("NVDA", "2026-01-15") 为例，状态如何流转：

  • ① 解析 NVDA 身份 + 回填上次 pending 收益 + 取历史教训
  • ② 四分析师串行产出 4 份报告
  • ③ 多空各一轮 → 研究经理判 Overweight
  • ④ 交易员出 Buy + 入场价/止损/3% 仓位
  • ⑤ 激进→保守→中性各一轮 → 组合经理拍 Buy
  • ⑥ 信号处理提取评级，落 pending 日志
  • ⑦ 下次运行回填 +4.5% / alpha +2.1% 并生成反思
