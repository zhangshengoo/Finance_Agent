01 三层定位 · 一条主线

A 是“决策 → 真实收益 → 写教训 → 下次召回”的闭环内核；回测是在历史上批量产生决策并度量收益的引擎；前端是只读渲染。三者全部复用 TradingAgents-CN 原生设施，不另起记忆/回测系统。

① 功能 A · 记忆写半边 内核
原生 reflect_and_remember() 早已存在但休眠（全仓库唯一引用是 main.py 一行注释）。补上它即闭合记忆环。
② 回测 / 交易模拟 M1
以个股分析决策（买/持/卖）为信号，从 Tushare 取前向真实收益，模拟单笔成交并算收益。
③ 前端只读面板 展示
成交表 + 指标卡 + 记忆教训卡，渲染 runner 导出的 KB 文件。本期不含触发按钮。

04 M1 单决策反思 · 执行流程

run_backtest.py --mode reflect 对每个 (snapshot, horizon) 的 7 步。不重跑引擎，只算收益 + 5 次轻量 reflect。

读 snapshot；若 status==failed 或 action==持有 且 confidence/risk==0.5/0.5（解析回退指纹）→ 跳过
查幂等台账，已 reflect 过 (ticker,date,horizon) → 跳过（除非 --force）
取价格 get_historical_data(ticker, date−3d, date+horizon+5d)，贴最近交易日取 entry/exit close；停牌或超 max-gap → 标不可评估并跳过
算签名收益：买入→+raw_ret，卖出→−raw_ret，持有→0/相对基准；生成 returns_losses 短串（含方向+%+天数+动作）
重建 curr_state（8 字段映射，None→''）
探活 DashScope → 建图 TradingAgentsGraph(config, memory_enabled=True) → ta.curr_state=state → ta.reflect_and_remember(returns_losses)
落盘：追加 trades.jsonl 一行 + 幂等台账一行

05 M2 · 前向纸面交易（组合模拟）设计 本期外 · 架构预留

M2 是 M1 的自然延伸：把“单笔决策反思”升级成随时间推进的组合模拟，产出净值曲线与组合指标，并在每次平仓时用真实持有期收益喂功能 A。关键取舍——只向前（forward paper-test），不重跑古早历史，从而方法论干净（真·样本外）且天然补齐“现状每票仅 1 个 snapshot”的序列缺口。

✅ 为什么 M2 向前而非向后？ 引擎 online_tools=True，舆情分析师抓的是当前新闻/情绪。在过去日期重跑（M3）会泄漏未来 → 前瞻偏差。M2 从今天起按节奏跑、向前累积，每个决策都是当时真实信息做出的样本外决策 → 净值曲线可信。这也是 M2 优先于 M3 的根本原因。

07 不变量合规

本地 only，绝不公网
前端 serve.sh 仅 localhost；本期无任何网络触发组件；外发前停确认。
raw/ 不可变
回测产物写 raw/data + raw/analysis，重跑=新 runid，绝不覆盖。
不直接写 wiki/
analysis → raw-intake → finance-ingest → wiki/reports。
ontology 只经 graph_append.py
1×Backtest 节点 + COVERS 边；不给成交/净值点建节点；先登记 schema.yaml。
永不编造
指标全部 sources[] 可溯；缺=null/[UNSOURCED]；sample badge 诚实标注。
DashScope 仅 Embedding
reflect 5 次对话走 anthropic:5678；嵌入走 dashscope；跑前探活防零向量。
引擎重跑很贵
M1 不重跑引擎；只 5 次轻量 reflect + 1 次 Tushare 区间拉取。
幂等防污染
sidecar 台账记 (ticker,date,horizon)；P1 先用 temp persist dir 验证再正式跑。
